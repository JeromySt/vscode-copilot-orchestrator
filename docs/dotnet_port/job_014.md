## Goal

Roslyn `IIncrementalGenerator` implementing **EV-GEN-*** rules from §3.4.6: every `[EventV(N)]`-attributed record participates in a generated migration table, generator emits `EventTypeRegistry`, requires a `IEventMigration<TFrom, TTo>` to exist for every adjacent version pair (build fails otherwise), and emits round-trip acceptance tests.

## Detailed implementation requirements

### Public API surface

```csharp
namespace AiOrchestrator.Models.Eventing;
[AttributeUsage(AttributeTargets.Class | AttributeTargets.Struct, AllowMultiple = false)]
public sealed class EventVAttribute : Attribute { public int Version { get; } public string EventTypeName { get; } public EventVAttribute(string eventTypeName, int version); }

public interface IEventMigration<TFrom, TTo> where TFrom : notnull where TTo : notnull { TTo Migrate(TFrom from); }

namespace AiOrchestrator.Eventing.Generators;

[Generator(LanguageNames.CSharp)]
public sealed class EventVersionGenerator : IIncrementalGenerator
{
    public void Initialize(IncrementalGeneratorInitializationContext context);
    // Emits:
    //   EventTypeRegistry.g.cs               — Frozen mapping (eventTypeName, version) -> Type
    //   EventMigrationGraph.g.cs             — verifies all adjacent version edges exist; build error if missing
    //   EventVersionAcceptanceTests.g.cs     — xUnit tests verifying every released version round-trips through the latest reader
}

internal sealed class MigrationTableEmitter { public string Emit(IReadOnlyList<EventVersionInfo> versions); }
internal sealed class AcceptanceTestEmitter { public string Emit(IReadOnlyList<EventVersionInfo> versions); }
internal sealed record EventVersionInfo(string EventTypeName, int Version, string FullyQualifiedTypeName);
```

### Behavioral invariants

1. **INV-1** (EV-GEN-1) — Generator scans all `[EventV(...)]` types in the compilation; emits a frozen registry mapping `(string eventTypeName, int version) -> Type`.
2. **INV-2** (EV-GEN-2) — For every event type with versions {1, ..., N}, generator requires a public `IEventMigration<V_{i}, V_{i+1}>` implementation in the compilation; missing implementations cause a build error with diagnostic `EVGEN001` pointing at both versions.
3. **INV-3** (EV-GEN-3) — Generator emits `EventVersionAcceptanceTests.g.cs` that, for each event type, deserializes a captured wire-format payload of every released version and asserts it round-trips through the migration chain to the latest version with no data loss (test data lives under `tests/dotnet/EventVersioning.Fixtures/<EventType>/v<N>.json`).
4. **INV-4** — Generator is `IIncrementalGenerator` and is deterministic across builds.
5. **INV-5** — Generator never reads files outside the compilation (no external state).
6. **INV-6** — Diagnostic `EVGEN001` (missing migration), `EVGEN002` (duplicate version on same eventTypeName), `EVGEN003` (gap in version sequence) all carry rule-doc URLs.
7. **INV-7** — Generator output files are tracked in source via `<EmitCompilerGeneratedFiles>true</EmitCompilerGeneratedFiles>` for review during code review.
