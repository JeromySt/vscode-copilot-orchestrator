# Job 016 — Audit Log

**Tier:** T4
**Producer id:** `t4-audit`
**Dependencies:** `t2-filesystem`, `t2-clock`, `t3-event-log`
**Estimated LOC:** ~1,100 production, ~1,500 tests
**Effort:** high (security-critical)

## Goal

Tamper-evident audit log per §3.5 + v1.4 hardenings. HMAC-SHA256 hash chain over append-only segments, Ed25519 signature per segment, embedded public-key header, key rotation with `KeyTransition` event, anchored at install time (`InstallAnchor`) and at every release-manifest update (`BuildKeyRolloverObserved`). v1.4 governance: daemon NEVER signs with build-key; daemon observes pre-signed `release-manifest.signed.json` produced by offline ceremony (job 39 + 43).

### Behavioral invariants

1. **INV-1** (KEY-ROT-1) — Each segment header carries `PrevSegmentHmac` + `PrevSegmentId`; `HmacChain.Compute(prevHmac, segmentBytes)` is the only chaining function.
2. **INV-2** (KEY-ROT-1) — Each segment is Ed25519-signed; the segment header embeds the public-key fingerprint of the signer for offline verification without external keystore.
3. **INV-3** (KEY-ROT-1) — Key rotation produces a `KeyTransition` record signed by BOTH the old and new keys (cross-signature); `ChainVerifier` rejects transitions missing cross-signatures (`AuditChainKeyTransitionMissingCrossSignature`).
4. **INV-4** (TRUST-ROOT-1) — Verification anchors at the install-time public key (`InstallAnchor`); chain is invalid if no path from `InstallAnchor.InitialAuditPubKey` to the segment signer through transitions.
5. **INV-5** (TRUST-ROOT-2) — Subsequent trust comes from `ReleaseManifest.TrustedAuditPubKeys`, which is signed by an offline root.
6. **INV-6** (TRUST-ROOT-4 v1.4) — Daemon code path NEVER calls `Ed25519Signer.Sign` with a build-key; the daemon ONLY observes `release-manifest.signed.json` and emits `BuildKeyRolloverObserved`. Test `TRUST_ROOT_4_DaemonNeverSignsWithBuildKey` proves zero call sites.
7. **INV-7** (TRUST-ROOT-6) — `VerifyMode.Strict` additionally checks each segment against an external Sigstore-style transparency log; fails with `AuditChainTransparencyLogMismatch`.
8. **INV-8** (TRUST-ROOT-7) — A daemon update that brings a new `ReleaseManifest` MUST NOT mark prior segments as tampered; `TRUST_ROOT_7_AuditChainSurvivesUpdate_NoFalseTamper` proves chain survives release-manifest version bump with new pubkeys when prior transitions are valid.
9. **INV-9** — All cryptographic operations use `System.Security.Cryptography` (HMAC-SHA256, Ed25519 via `Ed25519` from `NSec.Cryptography` or `BouncyCastle.Cryptography`); no homegrown crypto.
10. **INV-10** — Segment file format: magic `AIOA` + u16 version (= 1) + u32 record count + length-prefixed records + 32-byte HMAC + 64-byte Ed25519 signature + length-prefixed pubkey + length-prefixed header CBOR.
11. **INV-11** — Append failures roll the segment back atomically (write to `<segmentId>.tmp`, fsync, rename).
12. **INV-12** — `SegmentSeq` is monotonic; gap = `AuditChainSegmentSeqRegression`.
