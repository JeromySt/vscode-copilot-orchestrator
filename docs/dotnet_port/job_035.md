## Goal

MCP (Model Context Protocol) server implementation per §3.16. JSON-RPC over stdio (MCP-default) with tool list mirroring the existing TypeScript orchestrator's tool surface (scaffold_copilot_plan, add_copilot_plan_job(s), finalize_copilot_plan, get_copilot_plan_status, resume/pause/cancel/archive, reshape_copilot_plan, diagnose, plan-graph, etc.). Produces and consumes deterministic, schema-validated requests.

### Behavioral invariants

1. **INV-1** — JSON-RPC 2.0 framing: every request/response carries `jsonrpc: "2.0"`; non-conforming envelopes are rejected with `-32600 InvalidRequest`.
2. **INV-2** — Tool names match the existing TypeScript orchestrator surface exactly (used by VS Code extension and external clients).
3. **INV-3** — Each tool advertises a `JsonSchema` for its `InputSchema`; the server validates parameters BEFORE invocation and returns `-32602 InvalidParams` on schema violation.
4. **INV-4** — Tool invocations use source-gen `JsonTypeInfo<T>` for parameter binding (no reflection serializer).
5. **INV-5** — Tool invocations time out at `ToolInvokeTimeout` (default 60 s); timeout returns `-32000 Timeout`.
6. **INV-6** — Tool registry rejects duplicate tool names at startup (composition root catches it).
7. **INV-7** — Tools delegate to the daemon via the daemon RPC client (no direct file/git operations from MCP server).
8. **INV-8** — Stdio transport reads/writes line-delimited JSON-RPC framed by `Content-Length` header per MCP spec.
